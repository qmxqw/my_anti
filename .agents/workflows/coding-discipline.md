# 编码纪律：讨论 vs 编码

## 核心原则

**只有在用户明确表达"做"的意图时，才可以修改代码。**
技术讨论期间，只回答问题、分析可行性、提供方案建议，**绝不动代码**。

## 判断标准

### 🔴 讨论阶段（禁止改代码）

用户使用以下类型的表述时，属于讨论阶段：

- 询问类："能不能……？"、"……是怎么工作的？"、"如果……会怎样？"
- 确认类："确认一下……"、"这个方案没问题吗？"
- 探索类："有没有办法……？"、"技术上可行吗？"
- 分析类："为什么……？"、"区别是什么？"

**此阶段只能做：**
1. 阅读代码进行分析
2. 回答问题、解释原理
3. 提供方案建议（文字描述，不改代码）
4. 画出流程图或表格辅助说明

### 🟢 编码阶段（可以改代码）

用户使用以下类型的表述时，才进入编码阶段：

- 指令类："改一下……"、"把……改成……"、"实现……"、"加上……"
- 确认后的执行类："就按这个方案做"、"开始吧"、"动手"
- 明确的修改请求："帮我修复……"、"优化这段代码"

## 灰色地带处理

如果无法确定用户意图，**先问再做**：

> "你是想让我直接实现这个改动，还是先讨论一下方案？"

## 违规自检

在调用任何代码修改工具（write_to_file、replace_file_content、multi_replace_file_content）之前，
必须自问：**用户是在讨论还是在要求我改代码？**

## Git 命令执行规范

- git add / commit / log / status / diff 等本地操作：`WaitMsBeforeAsync` 设为 **10000**（最大值），确保同步完成
- 避免 git 命令因等待时间过短而被送入后台、再因轮询规则中断的问题
